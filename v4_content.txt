基圆游戏引擎 AI编程种子框架方案 V4.0 — 独立基圆 + 端口通信 + 积木组合 文档类型：AI编程指导 / 种子框架规范 版本：V4.0（基于V3.0增加独立基圆与通信系统） 日期：2026年6月 【最重要的理念：本项目结构分为底层种子框架+基圆代码，种子框架搭建好后就固定不该了，所有更改都在基圆代码中实现，不同类型的代码在不同类型的基圆里面实现】第一部分：基圆独立性与通信系统论证 本部分论证两个核心问题：第一，如何让每个基圆能够独立运行，像积木一样可以自由组合使用；第二，基圆之间的通信应该采用什么机制。这两个问题密切相关——独立性要求基圆不依赖特定的外部环境，而通信却需要基圆之间建立联系。如何在保持独立性的同时实现灵活的通信，是本方案的核心设计挑战。 1.1 独立基圆设计：积木化架构 独立基圆的核心理念是：每个基圆都是一个自包含的功能单元，拥有自己的代码、状态、资源和生命周期。它可以单独运行（如一只蚂蚁在屏幕上独自缩放），也可以与其他基圆组合（如蚂蚁+食物源+蚂蚁窝组成一个蚁群生态）。这种设计类似于微服务架构中的服务独立性原则——每个服务可以独立部署、独立运行、独立扩展，服务之间通过明确的接口通信。 独立性三原则： 第一，自包含代码。基圆的所有行为代码都在自己的文件/文件夹内，不依赖外部基圆的代码。引用其他基圆的代码通过import引用其导出的类型定义（如接口类型），而非直接调用其函数。第二，自管理状态。基圆的内部状态由自己维护，外部基圆不能直接修改，只能通过端口或事件发送请求。第三，自定义接口。基圆自己决定暴露哪些输入端口和输出端口，外部基圆只能通过这些端口与之交互。 积木组合机制： 基圆的积木组合分为两层：嵌套组合和连接组合。嵌套组合是父子关系——一个基圆内部可以包含子基圆，子基圆的坐标相对于父基圆，父基圆移动时子基圆跟随。例如一个“房子”基圆内部包含“门”“窗”“炉子”等子基圆。连接组合是平等关系——两个独立基圆通过端口线缆连接，形成数据流。例如一个“按钮”基圆的“按下”输出端口连接到“门”基圆的“打开”输入端口。两种组合方式可以自由混合使用。 1.2 通信方案对比论证 用户提出了三种可能的通信方案：通信基圆、地图基圆、其它方法。以下从架构质量、性能、可维护性、用户体验四个维度进行对比分析。 对比维度 方案A：通信基圆 方案B：地图基圆 方案C：端口+空间查询 架构质量 单点故障，所有消息经过中转站 仅解决空间查询，逻辑通信无解 分层解耦，无单点依赖 性能 每次通信经过中转，增加延迟 空间索引高效，但非通信解决方案 直达通信最快，空间查询用索引 可维护性 通信基圆自身需要维护和调试 地图基圆与其他基圆耦合度高 基圆不需要知道通信基础设施 用户体验 需要创建和配置通信基圆，增加操作步骤 需要创建地图基圆，学习成本高 端口可视化连线，直观易懂 独立性 基圆依赖通信基圆才能工作 基圆依赖地图基圆才能定位 基圆完全独立，通信是可选能力 方案A：通信基圆（Hub/Switchboard） 通信基圆作为所有基圆的消息中转站，所有基圆间的通信都经过它转发。优势是集中管理，可以在通信基圆中实现消息过滤、路由、记录等功能。劣势是显而易见的：第一，单点故障——通信基圆崩溃则所有基圆失联；第二，性能瓶颈——每次通信都经过中转，增加延迟；第三，违反独立性——基圆必须依赖通信基圆才能工作，无法独立运行。这与积木化架构的核心理念相矛盾。 方案B：地图基圆（Spatial Index） 地图基圆记录其他基圆在其内部的位置，方便基圆就近查找附近基圆。优势是空间查询高效，自然对应游戏世界的空间关系。劣势是：第一，仅解决空间查询问题，不解决逻辑通信问题（如按钮触发门打开，与空间位置无关）；第二，地图基圆与其他基圆耦合度高，必须先有地图基圆才能定位；第三，空间索引是引擎基础设施能力，不应该由一个游戏实体来承担。空间查询应该是CellContext的内置能力，而非一个独立基圆。 方案C：端口+空间查询（推荐采纳） 端口系统解决逻辑通信，空间查询API解决空间通信，事件总线解决松耦通信。三种机制分层解耦，各司其职，互不依赖。基圆完全独立——没有任何必须存在的中间基圆，通信是基圆的可选能力而非必要条件。一个基圆可以完全不与任何其他基圆通信而独立运行，也可以通过端口连接与其他基圆协作。这正是积木化架构的精髓——每块积木都能独立存在，组合后又能协同工作。 结论：采纳方案C，三层通信体系。 端口系统解决逻辑通信，空间查询API解决空间通信，事件总线解决松耦通信。不需要通信基圆或地图基圆作为中间层。 1.3 三层通信体系设计 基圆通信体系分为三层，每层解决不同的通信需求，基圆可以根据场景选择使用哪一层。三层之间互不依赖，可以独立使用，也可以组合使用。 第一层：端口通信（Port）——逻辑连接 端口通信解决基圆之间的明确数据流。每个基圆可以定义输入端口（接收数据）和输出端口（发送数据），端口有明确的类型定义。两个基圆之间通过线缆（Wire）连接输出端口到输入端口，形成明确的数据流向。这种方式类似于电路图中的连线或可视化编程中的节点连接，数据流向清晰可追踪。端口通信是同步的——当输出端口发送数据时，连接的输入端口立即收到。适用场景：按钮触发动作、传感器传递数据、开关控制设备、物品拾取通知。 第二层：空间查询（Spatial Query）——就近发现 空间查询解决基圆的就近发现问题。基圆不需要知道其他基圆的存在，只需要查询“我附近有谁”。这是通过CellContext的queryNearby方法实现的，底层使用引擎的空间哈希索引，性能极高。基圆可以根据查询结果决定自己的行为，例如蚂蚁发现附近有食物就走过去，发现附近有天敌就逃跑。空间查询是异步的——基圆在update函数中查询，结果立即返回。适用场景：生物寻路、追击逃跑、群体行为、物品拾取范围。 第三层：事件总线（Event Bus）——松耦广播 事件总线解决基圆之间的松耦通信。基圆发布事件时不知道谁会收到，订阅事件时不知道谁发出的。这种方式类似于无线电广播——发射器不知道接收器在哪里，接收器不知道发射器是谁。事件总线的优势是极度解耦——新增基圆可以订阅已有事件而无需修改发布者。劣势是调试困难——事件链难以追踪。适用场景：全局通知（天气变化、日夜切换）、游戏事件（得分、升级）、调试日志。 1.4 端口系统详细设计 端口系统是三层通信体系中最重要的一层，它让基圆之间的通信变得可视化、可追踪、可调试。每个基圆在init函数中声明自己的端口，外部基圆只能通过这些端口与之交互。 interface PortDef {   name: string;        // 端口名称，如'onPressed', 'signal'   type: string;        // 数据类型，如'boolean', 'number', '{x:number,y:number}'   direction: 'in' | 'out';  // 输入或输出   description?: string;     // 端口说明，供AI参考 } interface Wire {   id: string;   from: { cellId: string; port: string };  // 输出端   to: { cellId: string; port: string };    // 输入端 } 端口的工作流程：当基圆A的输出端口发送数据时，引擎查找所有从该端口出发的线缆，将数据传递到线缆另一端的输入端口。接收基圆的输入端口回调被触发，执行相应逻辑。如果一个输出端口连接多个输入端口，数据会广播到所有连接的输入端口。如果多个输出端口连接同一个输入端口，任何一个输出的数据都会触发该输入端口。 1.5 更新的CellContext接口 基圆独立运行是积木化架构的基础。每个基圆必须能够在没有任何其他基圆的情况下单独运行，展示自己的基本行为。这意味着：第一，基圆的init函数不能假设其他基圆已经存在；第二，基圆的update函数必须能够在没有外部输入的情况下正常工作；第三，基圆的render函数必须能够独立绘制自己。 独立运行的实现方式：引擎提供一个“独立测试模式”，在该模式下，只有被测试的基圆存在于世界中，没有其他基圆。基圆的queryNearby返回空数组，端口没有连接，事件没有订阅者。开发者可以在这个模式下验证基圆的基本行为是否正常，确保基圆不会因为缺少外部依赖而崩溃。这种测试方式类似于单元测试中的“孤立测试”——将被测试单元与所有依赖隔离，验证其自身逻辑的正确性。 1.7 通信示例：按钮开门 基于V3.0的关键提醒，V4.0进行以下优化： 优化1：基圆必须能独立运行 每个基圆的init/update/render函数不得假设其他基圆已经存在。queryNearby可能返回空数组，输入端口可能没有连接，事件可能没有订阅者。基圆必须优雅地处理这些情况，而不是崩溃或报错。引擎提供独立测试模式，开发者可以在该模式下验证基圆的独立性。 优化2：端口类型必须匹配 连接线缆时，输出端口的type必须与输入端口的type一致。例如输出端口类型为bool无法连接到输入端口类型number。引擎在创建线缆时进行类型校验，类型不匹配则拒绝连接并提示错误。这防止了运行时类型错误导致的崩溃。 优化3：端口连接可视化 端口连接必须在基圆面板和地图上可视化显示。在基圆面板的“连接”页签中，显示当前基圆的所有端口及其连接状态。在地图上，线缆以弹性曲线显示，从输出基圆连到输入基圆，点击线缆可查看连接详情或断开连接。这让通信关系一目了然，大大降低了调试难度。 优化4：空间查询的性能保障 queryNearby底层使用空间哈希索引，时间复杂度为O(k)（k为查询范围内的基圆数），不是O(n)。但如果基圆在update中每帧都调用queryNearby，仍然可能影响性能。建议：对于需要频繁查询的基圆，使用脉冲触发模式降低查询频率（如每0.5秒查询一次而非每帧），或使用onCollision回调代替主动查询。 优化5：代码段模式的端口定义 代码段模式下（多个基圆共享一个文件），每个基圆的端口定义必须在各自的@cell区块内完成。不允许在区块外定义端口，因为引擎只提取区块内的代码作为基圆的执行代码。如果多个基圆需要共享端口类型定义，应该将类型定义放在单独的types.ts文件中，各基圆通过import引用。 第二部分：更新后的种子框架方案
